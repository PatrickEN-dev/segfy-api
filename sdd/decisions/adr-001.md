# ADR-001 — Geração atômica do número da apólice em SQLite

**Status**: Aceita
**Data**: 2026-07-01
**Autores**: Architect Agent

---

## Contexto

O PDF do desafio exige que o número da apólice seja gerado automaticamente no formato **`SEG-YYYY-XXXX`**, onde:

- `SEG` é fixo,
- `YYYY` é o ano de emissão,
- `XXXX` é um sequencial (mínimo 4 dígitos, zero-padded).

O projeto persiste em **SQLite**. Diferente do PostgreSQL usado no projeto de referência, o SQLite **não tem `SEQUENCE`** como objeto de banco. Precisamos de outra estratégia que garanta:

1. **Atomicidade** — dois cadastros concorrentes não podem receber o mesmo número.
2. **Formato correto** — `SEG-{ano}-{contador zero-padded em 4 dígitos}`.
3. **Reset por ano** — a contagem reinicia em 1 no primeiro cadastro de cada novo ano.
4. **Portabilidade** — não depender de features exclusivas de um engine, para manter a possibilidade futura de migrar para PostgreSQL/SQL Server sem reescrever o domínio.

## Alternativas consideradas

### A) Contador em memória

Um `long` no `SqlitePolicyNumberSequence` incrementado em memória.

- Falha: perde estado a cada restart, perde ao rodar múltiplas instâncias.
- **Descartada.**

### B) `SELECT MAX(sequential) + 1`

Query implícita sobre `Policies`.

- Race condition: dois processos que leem o mesmo `MAX` inserem o mesmo número.
- Requer lock manual ou transação `SERIALIZABLE`, com custo desnecessário.
- Também dependeria de extrair o sequencial do formato `SEG-YYYY-XXXX`, o que é frágil (parser).
- **Descartada.**

### C) `ROWID` do SQLite

Usar o `ROWID` autoincrement da própria tabela `Policies`.

- Não permite reset por ano.
- Vaza detalhe interno (números pulariam se houver delete).
- **Descartada.**

### D) `AUTOINCREMENT` em coluna dedicada

Coluna `Sequential INTEGER PRIMARY KEY AUTOINCREMENT` na `Policies`.

- Ainda global, sem reset por ano.
- Persiste após delete (garantia do `AUTOINCREMENT`), o que é bom, mas não resolve o reset anual.
- **Descartada.**

### E) Tabela dedicada `PolicyNumberSequences(Year, LastValue)` — **ESCOLHIDA**

Uma tabela com uma linha por ano. Cada nova emissão:

1. Abre uma transação.
2. Busca (ou cria) a linha do ano atual.
3. Incrementa `LastValue`.
4. `SaveChanges` + `Commit`.
5. Retorna o novo `LastValue`.

- **Atômico** sob o modelo de concorrência do SQLite (writes serializados por padrão).
- **Reset natural por ano** — nova linha começa em 1.
- **Independente de engine** — funciona igual em SQL Server e PostgreSQL (a única diferença é o nome do provider EF).
- **Simples de auditar** — abrir o banco mostra o estado exato do contador em cada ano.

## Decisão

Adotar a alternativa **E**.

- Entidade de persistência: `PolicyNumberSequenceRow { int Year (PK), int LastValue }` em `Segfy.Infrastructure.Persistence`.
- Porta na Application: `IPolicyNumberSequence.NextForYearAsync(int year, CancellationToken)`.
- Implementação: `SqlitePolicyNumberSequence` em `Segfy.Infrastructure.Persistence.Sequences`, injetando o `SegfyDbContext` (mesmo `Scope` que o repositório — participa da mesma unidade de trabalho).
- Formato produzido: `PolicyNumber.Create(year, seq)` gera `SEG-{year:D4}-{seq:D4}`. Se `seq > 9999` o padding se expande (`10000`), sem quebrar leitura.

## Consequências

### Positivas

- Zero risco de colisão de número.
- Nada de código específico do SQLite — a mesma lógica migra para outros DBs.
- Reset anual gratuito: nova linha por ano começa em 1.
- Fácil de testar: `FakePolicyNumberSequence` cobre a mesma superfície com um `Dictionary<int, int>`.

### Negativas / trade-offs aceitos

- Um write extra por criação de apólice (upsert na sequência). Insignificante para o volume esperado.
- Transação explícita adicionada ao caminho de criação. Aceitável — é o preço da atomicidade.
- Se houver bulk import futuro, é necessário adaptar para reservar múltiplos números por chamada. Fora do escopo atual.

### Quando reconsiderar

- Se o projeto migrar para PostgreSQL exclusivo, avaliar substituir por `nextval` (mais barato, mesmo garantindo atomicidade).
- Se o volume passar a exigir milhares de emissões/segundo, considerar strategy híbrida (batch reservation ou snowflake-like).
